### What I did
1. Read `workshop/pipeline_scan/README.md` and the per-tool `workflow.yml` files.
2. Replaced the `.github/workflows/pipeline-scan.yml` placeholder with the **Claws** job (and copied `claws-config.yml` is already in repo).
3. Pushed to `dryrun-2026-05` and opened PR #7 against `main` to trigger `pipeline-orchestrator.yml`.
4. After observing Claws results, replaced the placeholder again with the **Zizmor** job and pushed.

### What happened
- **Claws run** ([run 25210635463](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25210635463)): 1 finding — `UnpinnedAction` on `ruby/setup-ruby@master` inside the **Claws workflow itself** (the file we had just copied in). Job failed with exit 1, blocking the rest of the orchestrator chain.
- **Zizmor run** ([run 25210691641](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25210691641)): job **passed (green)** even though the SARIF upload contained High-severity findings (`template-injection`, `unpinned-uses`). Orchestrator chain advanced; all downstream modules ran their placeholders successfully.

### Friction / didactic issues
1. **Module README is heavy on links, light on flow.** `workshop/pipeline_scan/README.md` lists tools, references and "extras" but never tells the new user the literal sequence: "open `claws/workflow.yml`, copy the `jobs:` block into `.github/workflows/pipeline-scan.yml`, push, open a PR." Beginners have to deduce that from the comment header inside `workflow.yml` plus the global `workshop/README.md`.
2. **The orchestrator is invisible to the user.** Nothing in the module README mentions that `pipeline-orchestrator.yml` exists, what triggers it (`pull_request` to `main`), or that subsequent modules will be skipped if pipeline-scan fails. A first-time user sees a single PR run scoreboard with many skipped jobs and no explanation.
3. **Claws produces a single finding — and it's in the file we just added.** The only violation is `UnpinnedAction` on `ruby/setup-ruby@master`, which is *inside* the Claws workflow we copied from `workshop/pipeline_scan/claws/workflow.yml`. Didactically this is confusing: the workshop tells you to copy something that is itself non-compliant. It's a weak way to demonstrate the value of Claws because the example doesn't show its breadth (no command-injection, risky-trigger, bulk-permissions, etc. examples in the rest of the repo).
4. **Zizmor "passes" while reporting High-severity issues.** The `zizmor-action@main` default behavior + `min-severity: "high"` does not make the job fail; the Run zizmor step exits 0 even with `template-injection` and `unpinned-uses` findings of severity High. The user sees a green checkmark and has no signal to investigate. This is the *opposite* of shift-left — it actively hides findings.
5. **Comment about `online-audits: false`** (`workshop/pipeline_scan/zizmor/workflow.yml:34`) refers to upstream issues without context. A new user will not know whether they should re-enable it later or what they lose by disabling it.
6. **Step 2 of `workshop/README.md`** says "follow the instructions in the `workflow.yml` file" — this places critical instructions inside YAML comments that aren't rendered on GitHub. They're easy to miss.
7. **No prerequisites listed.** The Claws workflow installs Ruby + shellcheck + a gem. None of this is mentioned in the README. A user trying to run this locally would be lost.

### Improvement proposals
- **High priority — make Zizmor actually fail on findings.** Either set `fail-on-findings: true` (or equivalent) in the action invocation, or add a follow-up step that parses the SARIF and exits non-zero. Otherwise the workshop teaches that "green = secure", which is the exact anti-pattern shift-left is supposed to fix.
- **High priority — pin `ruby/setup-ruby@master`** in `workshop/pipeline_scan/claws/workflow.yml` (or convert it into a guided "first finding to fix" example with explicit instructions in the README).
- **High priority — add an "Orchestrator overview" section** to `workshop/README.md` (not the per-module README) explaining: (a) what triggers the orchestrator, (b) the job dependency graph, (c) what "skipped" means in subsequent modules, (d) why a full PR is necessary (vs. `gh workflow run`).
- **Medium — flatten the README/workflow-comment split.** Move the "TO USE THIS JOB" instructions out of YAML comments and into the per-tool README block (or a single recipe per tool).
- **Medium — broaden the didactic surface for Claws.** Add 2-3 intentionally vulnerable example workflows under `workshop/pipeline_scan/examples/` that exercise more rules (`CommandInjection`, `BulkPermissions`, `RiskyTriggers`) so the user sees the tool's range — not just one Unpinned finding.
- **Low — add a "Prerequisites" section** in `workshop/pipeline_scan/README.md` listing dependencies (Ruby, shellcheck, claws-scan gem) for users who want to run locally.

For an attendee with no key, this tool produces a misleading "all green" — the exact opposite of the workshop's pedagogical goal. It would be safer to default the curriculum to osv-scanner (which works key-free) and present Dependency-Check as advanced/optional, with a clear explanation of why.
3. **CodeQL workflow uses a CLI-based approach** (`codeql database create` + `codeql database analyze`) instead of the higher-level `github/codeql-action/analyze@vX`. This is more "raw" and educational, but requires a follow-up `Add codeql to PATH` step and manual SARIF parsing. The README does not explain why the CLI path was chosen over the action-based approach. A short paragraph framing the trade-off would help.
4. **One CodeQL finding is thin for SAST didactic value.** The seeded vulnerability set has only the command-injection example. The workshop misses opportunities for teaching about XSS, prototype-pollution, taint-tracking explanations, etc. The README itself promises "SQL Injection, XSS, Command Injection, Path Traversal, Authentication Flaws…" as common findings, but the planted code only showcases command injection. Either expand the planted code, or trim the README claims.
5. **TODO comment in vulnerable code.** `simple-app.js:41` contains `// TODO: Tech debt - should use fs.readFile instead of shell command for security`. Funny in a dry-run, but a realistic teaching point would be: a TODO is not a security control. The README does not call this out.
6. **No mention that a passing scan does not imply a secure app.** CodeQL with `javascript-security-extended` still misses the hardcoded AWS key on lines 4-5 and the SSRF on `/fetch` (the latter has its own TODO). Module 3 (secrets-scan) will catch the keys, but the SSRF stays unfixed unless the user reads carefully. A sentence framing "SAST is a layer, not a guarantee" belongs in the README.
7. **Suggested SAST tool list is JS/Python-centric.** `workshop/code_scan/README.md:44` mentions ESLint/Bandit/Brakeman in a single inline note. For other ecosystems (Go, Java, Rust, .NET) the user gets no signposts. A short table in the README with one tool per ecosystem would be a high-value, low-cost change.

### Improvement proposals
- **High — fix the dangling `jobs.summary.outputs.result` reference** in `code-scan.yml`, either by documenting the expected job name in the placeholder or by including a tiny aggregation job.
- **High — recommend osv-scanner as the default SCA path** in the README, with Dependency-Check explicitly framed as "optional, requires NVD API key".
- **Medium — broaden the planted vulnerabilities in `code/`** to cover at least 2-3 SAST classes (e.g., XSS in the HTML response, SSRF in `/fetch` already partially planted but undetected, prototype pollution if Node version allows). This makes the post-fix delta more obvious and the lesson more memorable.
- **Medium — add an inline note in the README** that the orchestrator will skip downstream modules until each scan passes, so attendees know they need to fix to advance (TDD-style).
- **Low — add a short "SAST tools per ecosystem" table** to broaden the workshop's relevance beyond JavaScript.

3. Pushed; TruffleHog flagged the AWS access key. Fix: replaced the hardcoded `AKIA…` constants in `code/src/simple-app.js:4-5` with `process.env.*` reads.
4. Pushed; secrets-scan went green.

### What happened
- **First TruffleHog run** ([run 25211012964](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25211012964)) — failed with 1 unverified finding:
  ```
  Detector Type: AWS
  Raw result: AKIA2T2SJH6MS337PDWL
  File: code/src/simple-app.js:4
  ```
- **Post-fix run** ([run 25211070893](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25211070893)) — green; orchestrator advanced.

### Friction / didactic issues
1. **Filesystem scan only — git history is left untested.** The TruffleHog command is `trufflehog filesystem .` even though `fetch-depth: 0` is set on checkout. After the fix, the secret remains in commit `505952d` of git history. A real attacker would still find the leaked key. The workshop never demonstrates the harder lesson: *removing a secret from the file is not enough; the value must be considered compromised and rotated.* This is a prime spot to teach with `trufflehog git`.
2. **`--results=verified,unknown,unverified`** is correct for the planted (invalid) AKIA key, but the rationale lives only in a one-line YAML comment ("the workshop's didactic AKIA is invalid…"). New users won't grasp why this flag matters until they experiment. A short README paragraph titled "Why we accept unverified findings" would help.
3. **Gitleaks tool exists but is never wired in.** The README lists Gitleaks as the second tool, and the `workshop/secrets_scan/gitleaks/` directory contains a workflow snippet, but the workshop README doesn't push the user to actually try it after TruffleHog. With only one secret planted (the AKIA), there is no way for the user to discover that the two tools have different detector coverage.
4. **No cleanup-of-history exercise.** After the fix, history is left dirty. The README's "Security Checklist" item *"Git history cleaned of secrets"* is never demonstrated. An optional `git filter-repo` / `git rebase --interactive` mini-exercise (or at least a callout linking to GitHub's "remove sensitive data" docs) would reinforce this.
5. **Hardcoded `AWS_ACCESS_KEY_ID` constants are unused dead code.** The variables on `simple-app.js:4-5` are never referenced. After the fix, they remain dead `process.env.*` reads — still pollution. A more realistic teaching scenario would have the keys actually used by some axios call so the fix path requires *both* removing the literal *and* providing config via env or a secrets manager.
6. **The README is heavy on theory.** Lines 23-60 categorize secret types in detail (with footnotes) but never tell the user how to *add a secret* of their own to test detection coverage. A small "try it yourself" section ("add a fake `sk-…` OpenAI key to a comment, push, watch the scan") would convert passive reading into active exploration.

### Improvement proposals
- **High — switch to `trufflehog git file://. --since-commit HEAD~30`** (or `--all-objects`) so history coverage is part of the lesson. Then teach explicitly: *"The fix you just shipped does not erase the leak — rotate the key."*
- **High — add a short rotation/cleanup section** to the README, linking to `git filter-repo` or BFG and to GitHub's `removing-sensitive-data-from-a-repository` doc.
- **Medium — actually use the leaked key in code** (e.g., axios with an auth header), so the post-fix path forces the user to thread the secret through env vars / a secrets manager and they see realistic refactoring.
- **Medium — sequence Gitleaks as a follow-up tool comparison.** "TruffleHog caught X. Run Gitleaks on the same repo. Did it find the same? Different?" Even one paragraph in the README turns it from a list of links into a real exercise.
- **Low — surface the `--results=verified,unknown,unverified` rationale** as a README note, not just a YAML comment.

## Module 4 — Container Security Scanning

### What I did
1. Read `workshop/container_scan/README.md` and the Trivy snippet.
2. Replaced the placeholder in `.github/workflows/build-and-container-scan.yml` with the Trivy job.
3. Pushed; Trivy reported 13 CVEs in the `node:16.14.0-alpine` base image. Bumped to `node:22-alpine`.
4. Pushed; 1 HIGH CVE remained on `picomatch` bundled inside npm in the runtime image. Refactored the Dockerfile into a multi-stage build with a non-root distroless runtime (`gcr.io/distroless/nodejs22-debian12:nonroot`).
5. Pushed; 5 new CVEs surfaced in the distroless Debian base (`libssl3` CRITICAL + 4 HIGH). Switched runtime to `cgr.dev/chainguard/node:latest`.
6. Pushed; container-scan went green.

### What happened
- **Run 25211157388** — base `node:16.14.0-alpine`: 13 CVEs (Total HIGH+CRITICAL on `alpine 3.15.1`).
- **Run 25211237590** — base `node:22-alpine`: 1 HIGH CVE: `picomatch 4.0.3 → 4.0.4` (regex DoS) inside `usr/local/lib/node_modules/npm/...`.
- **Run 25211333070** — multi-stage with `gcr.io/distroless/nodejs22-debian12`: 5 CVEs surfaced (1 CRITICAL `libssl3 CVE-2026-31789`, 4 HIGH including `CVE-2026-28387` arbitrary code execution).
- **Run 25211412609** — multi-stage with `cgr.dev/chainguard/node:latest`: 0 HIGH/CRITICAL findings, orchestrator green end-to-end.

### Friction / didactic issues
1. **The README's TIP nailed the trajectory but stops one image short.** The TIP correctly predicts that bumping the base image clears OS CVEs but leaves bundled language-runtime libraries vulnerable. What it does *not* tell the user is that the next "obvious" fix — distroless — can itself be stale (Google's `distroless/nodejs22-debian12` shipped libssl3 with a CRITICAL openssl finding at the time of this run). The user can spend a substantial slice of the workshop chasing base-image churn. A short callout — "expect to compare base images, not just bump versions; consider Chainguard or other rolling-release providers" — would save a lot of time.
2. **`severity: "CRITICAL,HIGH"` + `ignore-unfixed: true` is a strong opinion that goes unexplained.** A user who hits 5 HIGH findings and reads the workflow yaml has no context for why MEDIUM/LOW are skipped, or what `ignore-unfixed` implies (i.e., Trivy hides CVEs without an upstream fix even if they are real). For a teaching pipeline this is fine, but the README should call it out.
3. **No SBOM step.** The README mentions Grype "uses multiple vulnerability databases and generates SBOM automatically" but the workshop never produces or attaches an SBOM. SBOMs are now compliance table-stakes (EO 14028, EU CRA). A `syft` step (5 lines) or simply enabling Trivy's SBOM output would be a high-impact addition for a 1-line cost.
4. **Dockerfile static analysis is missing.** The README's Security Checklist includes "Run as non-root", "multi-stage builds", "remove build tools" — but the workflow only scans the *image*, not the *Dockerfile*. Tools like `hadolint` or `dockerfilelint` would catch issues like a too-permissive `EXPOSE`, missing `USER`, or a hardcoded latest tag. The workshop is *more* about teaching practice than tooling; pairing image-scan with Dockerfile-lint reinforces both lessons.
5. **`HEALTHCHECK CMD` syntax is fragile across base images.** The original Dockerfile used `CMD node -e "..."` (shell form), which works on Alpine because there's a shell. Distroless and Chainguard runtimes have no shell, so the user has to switch to JSON-array form *and* know where the node binary lives. The workshop never explains this — users may produce a green scan with a broken HEALTHCHECK and not notice (no integration test exercises the healthcheck). At minimum, a comment or a tiny smoke test that runs the container and pings `/health` would close the loop.
6. **Grype is listed but not exercised.** Same Gitleaks-style issue as Module 3: the README invites comparison between Trivy and Grype but the workshop only wires up one. With only one scanner, the user never learns that scanners disagree on CVE coverage.

### Improvement proposals
- **High — recommend Chainguard (or equivalent rolling-release) base images** in the README as the path of least resistance to a clean container scan, with a pointer to distroless and a "but check that the openssl/CVE story is current" caveat.
- **High — pair Trivy with `hadolint`** (Dockerfile lint) in the same workflow. They run in parallel, are both fast, and together cover the README's full security checklist.
- **Medium — generate and upload an SBOM** (`trivy image --format cyclonedx`, or `syft`). Even if not enforced, it teaches the existence of SBOMs and gives an artifact to point at when a CVE breaks tomorrow.
- **Medium — add a comparative "now run Grype on the same image"** step or section, with a simple table for the user to fill in: what each tool found, what only one of them caught.
- **Low — explain the severity cutoff and `ignore-unfixed` flag** in the README, with a forward link to a triage / acceptance workflow (when is it OK to ignore a finding?).
- **Low — add a tiny smoke test** (`docker run -d image; curl localhost:3000/health`) so users verify the container actually boots after each refactor (HEALTHCHECK or USER changes are easy to break).

## Module 5 — IaC Security Scan

### What I did
1. Read `workshop/iac_scan/README.md` and the Checkov snippet.
2. Replaced the placeholder in `.github/workflows/iac-scan.yml` with the Checkov job (`directory: infra/`, `framework: terraform`).
3. Pushed; Checkov reported 2 hard failures plus 17 passes (with many `_INFO`-only / always-displayed checks). Fixed the two failing checks by scoping the ECS security group ingress to the VPC CIDR and to port 3000 only.
4. Pushed; iac-scan went green; the orchestrator is now end-to-end green for all in-scope modules.

### What happened
- **First Checkov run** ([run 25211546212](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25211546212)) — `Passed checks: 17, Failed checks: 2`. Both failures on `aws_security_group.ecs_tasks` (`infra/main.tf:115-148`):
  - `CKV_AWS_24` — security group allows ingress from `0.0.0.0/0` on port 22 (because `from_port=0, to_port=0` is interpreted as "all ports").
  - `CKV_AWS_25` — same, but for port 3389.
- **Post-fix run** ([run 25211618362](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25211618362)) — green; full orchestrator green.

### Friction / didactic issues
1. **The README mentions Trivy as the second IaC tool, but Trivy is not wired up.** Same one-tool-only problem as Modules 3 and 4. With only Checkov, the user never sees that scanners disagree (Checkov flagged 2 failures on this `infra/`; Trivy with default policies probably finds a different set, especially for things like missing log retention or KMS encryption that Checkov already counts as passes after recent changes).
2. **Checkov's "informational" output drowns the signal.** The log shows ~30+ `Check: …` lines with paths but only 2 actual `FAILED for resource`. A first-time reader has to manually scan to find what failed. The action invocation does not pass `--compact` or `--quiet` and does not filter to a specific severity. New users will be overwhelmed.
3. **The two planted misconfigurations are network-security only.** The README's "Common IaC Security Issues" section spans 5 categories (Access Control, Encryption, Network Security, Compliance & Governance, Supply Chain) but the actual planted infra only fails Network Security checks. The Checkov output even lists "informational" checks like `CKV_AWS_136 Ensure ECR repositories are encrypted using KMS` as passes — but only because the workshop's terraform conditionally adds KMS *if* a variable is set, and the variable defaults to empty. There's a real didactic chance to plant something in IAM (e.g., a wildcard `*` action) or encryption (an unencrypted RDS) but the workshop chose the simplest case.
4. **The fix path is whispered, not taught.** A clue exists at `infra/main.tf:113-114` ("Note from the SRE team: if you need to reference the VPC CIDR use this data:..."). Without this comment, a user might "fix" the rule by adding `cidr_blocks = ["10.0.0.0/8"]` (wrong constant, but plausible). The comment helps but the README never says "the planted issue is in the security group"; you only learn that by reading the scanner output. A short hint in the README ("the lab's `infra/` has at least one network-security finding — see the SRE note in `main.tf` for context") would lower the friction without giving away the answer.
5. **The pipeline never actually runs `terraform plan`.** All scanning is purely static. The `Terraform Plan RCE` reference in the README is excellent context but the workshop doesn't take the next step: running plan in the orchestrator to teach how a malicious provider can exploit it. Even a non-applying `terraform plan -refresh=false` step in `deploy-infrastructure.yml` (currently a dummy "Deploy Infrastructure" placeholder) would tie the lab back to the reference.
6. **`trivy:ignore:AVD-AWS-0104`** appears at `main.tf:136` for the HTTP egress rule but Trivy is never run, so this comment has no effect during the actual scan. Either drop the comment (since Checkov is doing the lifting) or wire Trivy IaC up.
7. **`infra/README.md` is empty / minimal.** The README at the root of `infra/` could serve as a "what does this stack model" — currently there is no orientation for a user reading the terraform cold.

### Improvement proposals
- **High — wire up Trivy IaC** as a parallel job inside `iac-scan.yml` so users see the two-tool comparison the README promises.
- **High — broaden planted misconfigurations** to span at least 3 categories (network, IAM with wildcard, encryption). One example each is enough; the current setup over-indexes on network security.
- **Medium — make Checkov's output consumable** by passing a smaller config (`soft_fail: false`, `--compact`, `--severity HIGH,CRITICAL`). The point is to fail fast on real issues, not flood the log.
- **Medium — add a `terraform plan` step (no apply)** in the orchestrator and reference the "Terraform Plan RCE" article as a teaching moment about supply chain in IaC providers.
- **Low — flesh out `infra/README.md`** with a 5-line description of what the stack provisions (ECS Fargate behind ALB), so users have a mental map before they read 250 lines of terraform.
- **Low — remove or activate the `trivy:ignore:` comment** at `main.tf:136` so it's not dead documentation.

  - **Medium** — when the workshop is ready to ship DAST, prefer a tool that runs entirely in-pipeline (e.g., `OWASP ZAP` baseline or `Nuclei` against `localhost`) rather than something that requires a publicly reachable target. The current orchestrator's reliance on an `app-url` is a weak link for an offline workshop.
  - **Low** — until DAST ships, add a 2-line "what would go here" preview in the README and a link to the placeholder so curious attendees can read ahead.

### AI Security Analysis (current Gemini integration considered outdated)
- **What's there today:** `workshop/ai_scan/gemini/workflow.yml` triggers on PR comments containing `/ai-review`. It uses `google-gemini/gemini-cli-action` and a free `GEMINI_API_KEY`. The flow is "comment on a PR → AI posts review comments". The module README is mostly conceptual (categories of issues AI can find, limitations).
- **Why this no longer reflects the state of the art (2026):**
  1. **The model and SDK have moved.** Gemini's pinned action and CLI assume an API surface several iterations old; current integrations use Tools / Function-calling / structured-output APIs. A user copying the snippet today often hits API-shape mismatches and quota errors.
  2. **The "review by chat comment" pattern is now a dead end.** Modern AI security review is agentic: a long-running agent that can read the diff, fetch the surrounding files, run static analyzers (CodeQL/Semgrep/etc.), and reason over their output before posting findings. A single LLM call against a diff is a 2023-style integration.
  3. **MCP changes the assumption set.** With MCP, the workshop could expose Trivy/Checkov/CodeQL as MCP servers and let the AI orchestrate them. That's the realistic 2026 pattern; the current single-prompt action does not teach it.
  4. **Single-vendor lock-in.** The module silently assumes Gemini. A workshop teaching DevSecOps practice should be model/vendor-agnostic and discuss trade-offs (privacy, cost, latency, security tool integration).
  5. **No threat model for the AI itself.** Prompt injection inside the diff (or the README of an upstream dep) is an active vector and not even mentioned.
- **Proposals (treat as a separate workshop redesign, not a quick fix):**
  - **High** — drop the single-action Gemini integration. Replace with a small dedicated module that demos *one* of:
    1. A GitHub Actions job that calls a CLI agent (Claude Code, Codex CLI, or Gemini CLI's modern equivalent) over a checked-out diff, with explicit prompt-injection guardrails on inputs.
    2. An MCP-based local demo (no GitHub Actions): user runs Claude Code or similar locally, server exposes Trivy/Checkov/Semgrep, and the model produces a triaged report. This is genuinely 2026-shaped.
  - **High** — add a section called "Threat model for AI in the pipeline": prompt injection in PR descriptions, untrusted issue comments triggering reviews, exfiltration via tool calls, cost / quota DoS.
  - **Medium** — keep the current `Limitations and Considerations` block — it ages well — but separate the "concept" content (which is fine) from the "implementation" content (which is stale).
  - **Medium** — make the AI module **explicitly the bonus track** it already claims to be in the top-level README. Today the module sits in the index alongside the others; users may assume it has the same maturity as Modules 1-5. Visually demoting it would set expectations.

### `dependency-check` tool (inside Module 2)
- **Reason not executed:** Skipped per agreement; the API-key-free `osv-scanner` covered SCA needs.
- **Observations from reading the snippet (`workshop/code_scan/sca/dependency_check/workflow.yml`):**

## Cross-cutting improvements

These observations apply to more than one module, or to the workshop as a whole.

### Onboarding and orientation
1. **The orchestrator is the most important file in the repo, and it is invisible.** A new user reads the top-level README and the per-module README but never the orchestrator. Yet the orchestrator dictates the entire experience: PR-driven runs, `needs:` chains that block on first failure, hard-coded URLs for the runtime stage. Add an "How the workflow runs" section to `workshop/README.md` with a 6-line dependency graph and a sentence on each chain edge.
2. **`workshop/README.md` step 2 is hostile to new users.** It says: *"For each tool within that module, follow the instructions in the `workflow.yml` file."* These instructions are buried in YAML comments inside the per-tool snippet. They are not rendered on GitHub. Move them into the per-module README so a reader who never opens the YAML still has the playbook.
3. **No "what to expect" on timing or feedback shape.** First-time users do not know that each push will queue a multi-job orchestrator that takes ~2 minutes, that finding output appears as logs *and* as Code Scanning alerts, or that GHAS findings persist across runs. A short FAQ at the top of `workshop/README.md` would save the typical "is it broken?" Slack ping.

### TDD-style flow needs to be named
4. The actual learning loop is **"add scanner → run → finding fails → fix code → run → green → next module"**. This is genuinely a TDD-style discipline. The README never names it. Calling it out explicitly ("This is red→green→refactor for security") would land the lesson better than the current "Run the workflows. Fix issues. Move on."

### Tool comparisons that are promised but not delivered
5. Modules 1, 3, 4 and 5 all list two tools but only wire up one in the orchestrator. The README of each module then asks the user to "try a different tool" without giving them a clear comparative exercise. **Either provide each module with a side-by-side comparison job (both scanners, separate findings), or drop the second tool entirely.** The current state is "promised and missing", which is the worst option.

### Severity and "green is not secure"
6. Several modules use severity cutoffs (Trivy `CRITICAL,HIGH`, Zizmor `min-severity high`, Trivy `ignore-unfixed: true`, Checkov implicit). The workshop never explicitly teaches the user **what these defaults hide**, **why they exist** (false positive volume, fix availability), or **how to triage exceptions**. A short module-zero-style "How to read security findings" appendix would tie the modules together.

### Image and base-version churn
7. Modules 2 (osv-scanner) and 4 (Trivy) both produce findings driven by the **passage of time** between the workshop being authored and being run. Both need a maintenance cadence (weekly or monthly bumps) or they will surface stale CVEs as the headline finding instead of the planted didactic ones. Consider a small cron job that opens a PR to bump base images and `axios` whenever the planted finding is no longer the *primary* one.

### Workflow file consistency
8. Some workflow scaffolds reference jobs that do not exist (`code-scan.yml` line 7 → `jobs.summary.outputs.result`). Others use a single job (`pipeline-scan`, `iac-scan`). The placeholder/job-name contract should be uniform across all scaffolds so a user copy-pasting a snippet does not get a dangling output reference.

### Pinning policy
9. The workshop pins most actions to SHA + version comment, which is best practice — but `ruby/setup-ruby@master` (Module 1, Claws snippet) and `zizmorcore/zizmor-action@main` and `cgr.dev/chainguard/node:latest` slip through. This is exactly the kind of inconsistency Claws / Zizmor exist to flag, and the workshop should hold itself to its own standard.

### Documentation hygiene
10. `infra/README.md` (and probably others under `code/`, `workshop/`) are minimal or empty. A 3-5 line "what does this directory contain" header on each top-level subdirectory would lower the cold-read cost dramatically.
11. Several `# TODO:` comments are planted in `code/src/simple-app.js` (e.g., the SSRF on `/fetch`, the original command injection). After the workshop's fix path, the TODOs become misleading or stale. Either the workshop should have an explicit pass to clean them up, or it should teach "TODOs are not security controls".

### Where the "perfect pipeline" ends
12. The top-level README states the goal as building a "perfect pipeline". The workshop covers shift-left (modules 1-5, runtime, dast) and AI review, but skips:
    - **Supply chain attestations** (Sigstore / SLSA / cosign image signing). With CRA / EO-14028 / SLSA Level 2 increasingly mandated, this is a glaring omission for a workshop that calls itself "perfect".
    - **Pull request enforcement** (required checks, branch protection, CODEOWNERS). The workshop builds a pipeline but never tells the user how to make it *enforced*.
    - **Triage / exception workflow.** Findings cannot all be fixed. A short module on `# checkov:skip=`, `nosec`, SARIF suppressions, and the human review trail when a finding is *accepted* would close the loop.
