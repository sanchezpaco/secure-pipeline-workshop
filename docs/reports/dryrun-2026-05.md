## Module 1 — Pipeline Security Scan

### What I did
1. Read `workshop/pipeline_scan/README.md` and the per-tool `workflow.yml` files.
2. Replaced the `.github/workflows/pipeline-scan.yml` placeholder with the **Claws** job (and copied `claws-config.yml` is already in repo).
3. Pushed to `dryrun-2026-05` and opened PR #7 against `main` to trigger `pipeline-orchestrator.yml`.
4. After observing Claws results, replaced the placeholder again with the **Zizmor** job and pushed.

### What happened
- **Claws run** ([run 25210635463](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25210635463)): 1 finding — `UnpinnedAction` on `ruby/setup-ruby@master` inside the **Claws workflow itself** (the file we had just copied in). Job failed with exit 1, blocking the rest of the orchestrator chain.
- **Zizmor run** ([run 25210691641](https://github.com/sanchezpaco/secure-pipeline-workshop/actions/runs/25210691641)): job **passed (green)** even though the SARIF upload contained High-severity findings (`template-injection`, `unpinned-uses`). Orchestrator chain advanced; all downstream modules ran their placeholders successfully.

### Friction / didactic issues
1. **Module README is heavy on links, light on flow.** `workshop/pipeline_scan/README.md` lists tools, references and "extras" but never tells the new user the literal sequence: "open `claws/workflow.yml`, copy the `jobs:` block into `.github/workflows/pipeline-scan.yml`, push, open a PR." Beginners have to deduce that from the comment header inside `workflow.yml` plus the global `workshop/README.md`.
2. **The orchestrator is invisible to the user.** Nothing in the module README mentions that `pipeline-orchestrator.yml` exists, what triggers it (`pull_request` to `main`), or that subsequent modules will be skipped if pipeline-scan fails. A first-time user sees a single PR run scoreboard with many skipped jobs and no explanation.
3. **Claws produces a single finding — and it's in the file we just added.** The only violation is `UnpinnedAction` on `ruby/setup-ruby@master`, which is *inside* the Claws workflow we copied from `workshop/pipeline_scan/claws/workflow.yml`. Didactically this is confusing: the workshop tells you to copy something that is itself non-compliant. It's a weak way to demonstrate the value of Claws because the example doesn't show its breadth (no command-injection, risky-trigger, bulk-permissions, etc. examples in the rest of the repo).
4. **Zizmor "passes" while reporting High-severity issues.** The `zizmor-action@main` default behavior + `min-severity: "high"` does not make the job fail; the Run zizmor step exits 0 even with `template-injection` and `unpinned-uses` findings of severity High. The user sees a green checkmark and has no signal to investigate. This is the *opposite* of shift-left — it actively hides findings.
5. **Comment about `online-audits: false`** (`workshop/pipeline_scan/zizmor/workflow.yml:34`) refers to upstream issues without context. A new user will not know whether they should re-enable it later or what they lose by disabling it.
6. **Step 2 of `workshop/README.md`** says "follow the instructions in the `workflow.yml` file" — this places critical instructions inside YAML comments that aren't rendered on GitHub. They're easy to miss.
7. **No prerequisites listed.** The Claws workflow installs Ruby + shellcheck + a gem. None of this is mentioned in the README. A user trying to run this locally would be lost.

### Improvement proposals
- **High priority — make Zizmor actually fail on findings.** Either set `fail-on-findings: true` (or equivalent) in the action invocation, or add a follow-up step that parses the SARIF and exits non-zero. Otherwise the workshop teaches that "green = secure", which is the exact anti-pattern shift-left is supposed to fix.
- **High priority — pin `ruby/setup-ruby@master`** in `workshop/pipeline_scan/claws/workflow.yml` (or convert it into a guided "first finding to fix" example with explicit instructions in the README).
- **High priority — add an "Orchestrator overview" section** to `workshop/README.md` (not the per-module README) explaining: (a) what triggers the orchestrator, (b) the job dependency graph, (c) what "skipped" means in subsequent modules, (d) why a full PR is necessary (vs. `gh workflow run`).
- **Medium — flatten the README/workflow-comment split.** Move the "TO USE THIS JOB" instructions out of YAML comments and into the per-tool README block (or a single recipe per tool).
- **Medium — broaden the didactic surface for Claws.** Add 2-3 intentionally vulnerable example workflows under `workshop/pipeline_scan/examples/` that exercise more rules (`CommandInjection`, `BulkPermissions`, `RiskyTriggers`) so the user sees the tool's range — not just one Unpinned finding.
- **Low — add a "Prerequisites" section** in `workshop/pipeline_scan/README.md` listing dependencies (Ruby, shellcheck, claws-scan gem) for users who want to run locally.
